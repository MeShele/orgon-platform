# Бизнес-роли и процессы ORGON Platform

**Дата:** 13 февраля 2026  
**Версия:** 1.0  
**Статус:** Финальная версия  

---

## 🎯 Обзор ролевой модели

ORGON Platform работает как многоуровневая B2B WaaS платформа, где каждый уровень имеет свои роли, полномочия и бизнес-процессы. Система поддерживает multi-tenancy архитектуру для обслуживания 170+ криптообменников Кыргызстана.

### Иерархия ролей

```
SuperAdmin (ASYSTEM) 
    ↓
PlatformAdmin (ASYSTEM техники)
    ↓
CompanyAdmin (руководитель обменника)
    ↓
CompanyOperator (сотрудник обменника)
    ↓
CompanyAuditor (аудитор/бухгалтер)
    ↓
EndUser (клиент обменника)
```

## 👤 Детализация ролей

### 1. SuperAdmin (ASYSTEM) 🔱

**Кто это:** Основатели и руководство ASYSTEM  
**Зона ответственности:** Стратегическое управление платформой, монетизация, партнерские отношения с KAZ.ONE

#### Права доступа:
- **Полный доступ** ко всем данным всех организаций
- **Управление тарифными планами** (A, B, C)
- **Биллинг и финансовая отчетность** (revenue share с KAZ.ONE)
- **Создание/блокировка организаций**
- **Системные настройки** (интеграции, белый список IP)
- **Мониторинг** performance всей платформы

#### Доступные страницы:
- **Dashboard:** агрегированная статистика всех клиентов
- **Organizations:** управление обменниками (создание, блокировка, тарифы)  
- **Billing:** revenue отчеты, абонентская плата, взаиморасчеты с KAZ.ONE
- **Analytics:** кросс-организационная аналитика, benchmark клиентов
- **Compliance:** regulatory отчеты для Финнадзор КР по всем обменникам
- **Support:** все тикеты, эскалация проблем
- **Settings → Platform:** глобальные настройки (Safina API, fiat gateways)

#### Типичные сценарии:
- **Утренний ритуал:** проверка ночных incidents, revenue dashboard, статус KAZ.ONE интеграции
- **Онбординг нового обменника:** создание организации → настройка тарифа → проверка KYB документов → активация
- **Регуляторная проверка:** генерация сводного отчета по всем 170+ обменникам для Финнадзор
- **Биллинг цикл (месячный):** расчет комиссий, выставление счетов, revenue share с KAZ.ONE

### 2. PlatformAdmin (ASYSTEM техники) ⚙️

**Кто это:** Техническая команда ASYSTEM (DevOps, Support L2/L3)  
**Зона ответственности:** Техническая поддержка, monitoring, troubleshooting

#### Права доступа:
- **Read-only доступ** к техническим данным всех организаций
- **Управление support тикетами** (L2/L3)
- **Мониторинг системы** (uptime, performance)
- **Техническая конфигурация** (API endpoints, webhooks)
- **Без доступа** к финансовым данным и биллингу

#### Доступные страницы:
- **Dashboard:** system health, API status, uptime metrics
- **Support:** техническая поддержка, эскалация от L1
- **Analytics:** только технические метрики (response times, error rates)
- **Audit:** полный технический лог (API calls, errors, performance)
- **Settings → System:** мониторинг Safina API, database connections

#### Типичные сценарии:
- **Incident response:** обработка алертов о недоступности API, восстановление сервисов
- **Support L2:** решение сложных техничских проблем клиентов
- **Performance monitoring:** оптимизация медленных запросов, scaling решения

### 3. CompanyAdmin (руководитель обменника) 🏢

**Кто это:** Директор/владелец криптообменника  
**Зона ответственности:** Стратегическое управление обменником, compliance, финансы

#### Права доступа:
- **Полный доступ** к данным своей организации
- **Управление пользователями** своей организации (создание CompanyOperator, CompanyAuditor)
- **Настройки белого лейбла** (брендинг, логотип, домен)
- **Финансовая отчетность** (биллинг, комиссии, P&L)
- **Compliance настройки** (AML/KYC политики)
- **Без доступа** к данным других обменников

#### Доступные страницы:
- **Dashboard:** сводка по своему обменнику (объемы, клиенты, доходы)
- **Wallets:** все кошельки обменника (hot, warm, corporate)
- **Transactions:** история всех операций
- **Analytics:** бизнес аналитика (top клиенты, profitable направления)
- **Billing:** счета от ASYSTEM, история платежей, комиссии
- **Compliance:** AML отчеты, suspicious transactions, регуляторные формы
- **Users:** управление сотрудниками (создание, блокировка, права)
- **Settings → Organization:** брендинг, fiat gateways, API keys
- **Support:** создание тикетов, история обращений

#### Типичные сценарии:
- **Стратегическое планирование:** анализ performance, определение growth направлений
- **Compliance ревью:** ежемесячная проверка AML alerts, подготовка отчетов для Финнадзор
- **HR:** найм новых операторов, настройка их доступов
- **Финансовый контроль:** контроль расходов на ASYSTEM услуги, оптимизация тарифа

### 4. CompanyOperator (сотрудник обменника) 👨‍💼

**Кто это:** Ежедневные операторы обменника (кассиры, traders)  
**Зона ответственности:** Ежедневные операции, работа с клиентами

#### Права доступа:
- **Операционный доступ** к кошелькам и транзакциям
- **Создание транзакций** (с лимитами)
- **Подписание простых операций** (до определенной суммы)
- **Работа с клиентами** (KYC, поддержка)
- **Без доступа** к настройкам организации и крупным суммам

#### Доступные страницы:
- **Dashboard:** операционная сводка (pending transactions, alerts)
- **Wallets:** просмотр балансов, создание новых кошельков для клиентов
- **Transactions:** создание, отправка, отслеживание статуса
- **Contacts:** адресная книга клиентов
- **Scheduled:** настройка регулярных платежей
- **Support:** создание тикетов, база знаний
- **Settings → Profile:** личные настройки, 2FA

#### Типичные сценарии:
- **Обслуживание клиента:** создание кошелька → прием фиата → покупка криптовалюты → отправка клиенту
- **Ежедневные операции:** проверка pending signatures, обработка входящих платежей
- **KYC процесс:** верификация новых клиентов, загрузка документов

### 5. CompanyAuditor (аудитор/бухгалтер) 📊

**Кто это:** Аудитор или главный бухгалтер обменника  
**Зона ответственности:** Финансовый контроль, compliance, отчетность

#### Права доступа:
- **Read-only доступ** ко всем финансовым операциям
- **Генерация отчетов** (financial, compliance, regulatory)
- **Аудит логи** (все действия сотрудников)
- **Без права создания** операций
- **Специальные права** на экспорт данных

#### Доступные страницы:
- **Dashboard:** финансовая сводка (PnL, комиссии, оборот)
- **Transactions:** read-only режим, расширенная фильтрация
- **Analytics:** финансовые отчеты, регуляторная аналитика
- **Audit:** полный audit trail всех операций сотрудников
- **Compliance:** генерация отчетов для Финнадзор, AML мониторинг
- **Export:** выгрузка данных в Excel/PDF для отчетности

#### Типичные сценарии:
- **Месячный closing:** сверка оборотов, генерация P&L отчета
- **Regulatory reporting:** подготовка форм для Финнадзор КР  
- **Internal audit:** проверка действий операторов, выявление нарушений
- **Tax compliance:** подготовка налоговых отчетов

### 6. EndUser (клиент обменника) 💳

**Кто это:** Физические лица — клиенты обменника  
**Зона ответственности:** Управление своими средствами через white-label интерфейс

#### Права доступа:
- **Доступ только к своим** кошелькам и транзакциям
- **Создание операций** (в рамках KYC лимитов)  
- **Просмотр истории** своих операций
- **Без доступа** к данным других клиентов

#### Доступные страницы (в white-label интерфейсе):
- **My Wallets:** личные кошельки
- **Send Money:** отправка криптовалют
- **Transaction History:** история операций
- **Profile:** KYC статус, лимиты, персональные данные

#### Типичные сценарии:
- **Покупка криптовалюты:** оплата фиатом → получение crypto в кошелек
- **Отправка денег:** перевод криптовалюты другому человеку
- **Cash-out:** конвертация crypto → фиат → вывод на банковскую карту

## 🔄 Ключевые бизнес-процессы

### 1. Регистрация компании → KYB → Настройка

```
SuperAdmin действия:
├─ Получение заявки от обменника
├─ KYB проверка (лицензия ПУВА, уставные документы)
├─ Создание Organization в ORGON
├─ Выбор тарифного плана (A/B/C)
├─ Настройка начальных лимитов
└─ Активация аккаунта

CompanyAdmin действия:
├─ Получение credentials (email/password)
├─ First login → обязательная настройка 2FA
├─ Настройка организации (брендинг, контакты)
├─ Создание первых кошельков
└─ Интеграция fiat gateway (банк/платежная система)
```

**Временные рамки:** 2-3 рабочих дня  
**Критерии успеха:** Первая успешная crypto транзакция

### 2. Создание кастодиального кошелька

#### Одноподписной кошелек (простой)

```
CompanyOperator:
  ↓
1. Wallets → "Создать кошелек"
  ↓  
2. Выбор сети (Bitcoin/Ethereum/Tron/BNB)
  ↓
3. Указание метки (например, "BTC Hot Wallet #1")
  ↓
4. Выбор типа: Single-sig
  ↓
5. [Backend] POST /api/wallets
   {
     "network": 1,
     "wallet_type": 0,
     "label": "BTC Hot Wallet #1"
   }
  ↓
6. [Safina API] POST /ece/newWallet
   {
     "network": 1,
     "wtype": 0
   }
  ↓
7. Получение wallet_name от Safina
  ↓
8. Сохранение в БД + показ адреса пользователю
```

**Время выполнения:** 10-15 секунд  
**Состояния UI:** Loading → Success (показ адреса) / Error

#### Мультиподписной кошелек (complex)

```
CompanyAdmin или старший CompanyOperator:
  ↓
1. Wallets → "Создать мультиподпись"
  ↓
2. Выбор сети
  ↓
3. Настройка схемы подписей (например, 2 из 3)
  ↓
4. Указание списка подписантов (slist)
   [
     "ec_address_1", 
     "ec_address_2", 
     "ec_address_3"
   ]
  ↓
5. [Backend] POST /api/wallets
   {
     "network": 1,
     "wallet_type": 1,
     "slist": ["addr1", "addr2", "addr3"],
     "min_signatures": 2
   }
  ↓
6. [Safina API] POST /ece/newWallet
   {
     "network": 1,
     "wtype": 1,
     "slist": ["addr1", "addr2", "addr3"]
   }
  ↓
7. Wallet создан → уведомление подписантов
```

**Время выполнения:** 30-60 секунд  
**Критический момент:** Корректная настройка slist (список EC адресов подписантов)

### 3. Пополнение кошелька (входящие транзакции)

```
Внешняя система/пользователь отправляет crypto
  ↓
1. Транзакция попадает в blockchain
  ↓
2. [Safina] обнаруживает входящую транзакцию
  ↓
3. [Safina] отправляет webhook на ORGON
   POST /api/webhooks/incoming_transaction
   {
     "wallet_name": "btc_wallet_001",
     "token": "BTC",
     "amount": "0.05",
     "tx_hash": "abc123...",
     "confirmations": 1
   }
  ↓
4. [ORGON Backend] обновляет баланс в БД
  ↓
5. [WebSocket/SSE] уведомление фронтенда
  ↓
6. [Frontend] обновляет баланс в реальном времени
```

**Критические моменты:**
- **Webhook безопасность:** HMAC подпись для верификации
- **Idempotency:** обработка дублирующих webhook'ов
- **Real-time updates:** WebSocket/SSE для мгновенного обновления UI

### 4. Отправка транзакции (одноподписной)

```
CompanyOperator:
  ↓
1. Transactions → "Отправить"
  ↓
2. Заполнение формы:
   - Кошелек отправителя
   - Адрес получателя 
   - Сумма
   - Токен (BTC/ETH/USDT/etc.)
  ↓
3. Предварительная проверка:
   - Достаточность баланса
   - Корректность адреса получателя
   - AML/KYC проверки
  ↓
4. [Backend] POST /api/transactions
   {
     "from_wallet": "btc_wallet_001",
     "to_address": "1A1zP1eP5Q...",
     "amount": "0.01",
     "token": "BTC"
   }
  ↓
5. [Safina API] POST /ece/tx
   {
     "token": "BTC",
     "to": "1A1zP1eP5Q...",
     "value": "1000000",  // satoshi
     "wallet": "btc_wallet_001"
   }
  ↓
6. Safina возвращает tx_unid
  ↓
7. Для single-sig: автоматический broadcast в blockchain
  ↓
8. [Frontend] показ статуса "Pending" → "Confirmed"
```

**Время выполнения:** 
- **Bitcoin:** 10-60 минут (зависит от network congestion)
- **Ethereum:** 1-5 минут
- **Tron:** 3-30 секунд

### 5. Мультиподписная транзакция (сложный процесс)

```
CompanyOperator инициатор:
  ↓
1. Создание транзакции (аналогично single-sig)
  ↓
2. [Safina API] создает транзакцию со статусом "pending signatures"
  ↓
3. [ORGON] отправляет уведомления всем подписантам:
   - Email уведомление
   - Push notification в мобильном приложении
   - Webhook для внешних систем
  ↓

Каждый подписант:
  ↓
4. Signatures → "Pending" список
  ↓
5. Просмотр деталей транзакции:
   - Сумма, получатель, инициатор
   - AML/risk score
   - Комментарии инициатора
  ↓
6. Принятие решения: "Approve" или "Reject"
  ↓

Approve flow:
7a. [Frontend] POST /api/transactions/{unid}/sign
    {
      "action": "approve",
      "comment": "Approved by CFO"
    }
  ↓
8a. [Safina API] POST /ece/tx_sign/{tx_unid}
  ↓
9a. Safina регистрирует подпись
  ↓

Reject flow:
7b. [Frontend] POST /api/transactions/{unid}/sign
    {
      "action": "reject", 
      "reason": "Suspicious recipient"
    }
  ↓
8b. [Safina API] POST /ece/tx_reject/{tx_unid}
  ↓
9b. Транзакция отклонена → уведомления всем участникам

Approval threshold достигнут:
  ↓
10. [Safina] автоматически отправляет транзакцию в blockchain
  ↓  
11. [Webhook] ORGON получает уведомление о broadcast
  ↓
12. [Frontend] статус меняется на "Broadcasting" → "Confirmed"
```

**SLA метрики:**
- **Время ожидания подписей:** до 24 часов
- **Эскалация:** если нет ответа 12 часов → email напоминание
- **Auto-reject:** если нет ответа 48 часов (опционально)
