## 2. Маппинг требований на функции ORGON

### 2.1 Раздельное хранение активов (п.5 Положения 1, Постановление 625)

**Требование:** Совместное хранение ВА двух и более клиентов **не допускается**. Каждый клиент = отдельный кошелёк.

**Покрытие ORGON:**

| Компонент | Статус | Реализация |
|-----------|--------|------------|
| Раздельные кошельки через Safina | ✅ Покрыто | `POST /ece/newWallet` — каждый кошелёк привязан к организации |
| Multi-tenancy с RLS | ✅ Покрыто | PostgreSQL Row-Level Security изолирует данные тенантов |
| Учёт принадлежности кошельков | ✅ Покрыто | `wallets.org_id` → каждый кошелёк привязан к организации |
| Запрет cross-tenant операций | ✅ Покрыто | RLS policies блокируют доступ к чужим данным |
| Аудит разделения | ⚠️ Нужно доработать | Нужен отчёт: "все кошельки разделены по организациям" |

**Что нужно доработать:**
- Добавить automated check: ни один кошелёк не принадлежит > 1 организации
- Отчёт для регулятора: маппинг "организация → кошельки → адреса"

### 2.2 AML/KYC/KYT мониторинг (Положения 1, 3; Постановление 625)

**Требование:** Обязательный мониторинг транзакций, идентификация и верификация клиентов.

**Покрытие ORGON:**

| Компонент | Статус | Реализация |
|-----------|--------|------------|
| KYC при регистрации | ✅ Базово | Email + пароль, JWT auth |
| KYB для организаций | ⚠️ Частично | Создание организации есть, верификация документов — нет |
| KYT мониторинг | ⚠️ Через Safina | Safina CCC для анализа транзакций |
| AML скрининг | ⚠️ Через Safina | Safina AML модули |
| Suspicious Activity Reports | ❌ Не реализовано | Нужно: автоматические SAR при risk > threshold |
| Transaction monitoring rules | ⚠️ Базово | Compliance routes есть, но правила не настроены |

**Что нужно доработать:**
1. **KYB процесс:** загрузка документов (лицензия, учредительные), review workflow, статусы (pending → verified → rejected)
2. **KYC расширенный:** идентификация физ.лиц (паспорт, селфи), автоматическая верификация
3. **SAR (Suspicious Activity Reports):** автоматическая генерация при:
   - Транзакция > $10,000 (или эквивалент)
   - Множественные мелкие транзакции (structuring)
   - Транзакции с высокорисковых адресов (blacklists)
4. **Transaction monitoring rules engine:** настраиваемые правила для каждой организации

### 2.3 Запрет Private Wallets без идентификации (п.24 Положения 3, Постановление 625)

**Требование:** Запрещено использовать кошельки повышенной конфиденциальности без идентификации держателей.

**Покрытие ORGON:**

### 2.4 Процентный скоринг рисков (п.24 Положения 3)

**Требование:** Запрещены операции с ВА/кошельками с высоким уровнем рисков по скорингу.

**Покрытие ORGON:**

| Компонент | Статус | Реализация |
|-----------|--------|------------|
| Risk scoring API | ⚠️ Через Safina | CCC (Crypto Compliance Console) |
| Auto-block high risk | ❌ Не реализовано | Нужно: автоблокировка при risk > 80% |
| Risk dashboard | ⚠️ Базово | Compliance page есть, но без live data |
| Risk alerts | ❌ Не реализовано | Нужно: Telegram/email уведомления |

**Реализация (Phase 2):**
```python
# backend/services/compliance_service.py
async def check_transaction_risk(tx_data):
    """Проверка риска транзакции через Safina CCC"""
    risk_score = await safina_client.get_risk_score(tx_data.to_address)
    
    if risk_score > 80:
        await block_transaction(tx_data.id, reason="high_risk")
        await notify_admin(f"Blocked TX {tx_data.id}: risk {risk_score}%")
        await create_sar(tx_data, risk_score)
        return {"status": "blocked", "risk": risk_score}
    
    if risk_score > 50:
        await flag_transaction(tx_data.id, reason="medium_risk")
        return {"status": "flagged", "risk": risk_score}
    
    return {"status": "clear", "risk": risk_score}
```

### 2.5 Учёт и отчётность для регулятора (Приказы 579п, 580)

**Требование:** Обязательная отчётность ПУВА для Финнадзор.

**Покрытие ORGON:**

| Компонент | Статус | Реализация |
|-----------|--------|------------|
| Учётная система | ✅ Базово | transactions, wallets, organizations в БД |
| Отчёты для клиентов | ⚠️ Частично | Analytics page есть, export — нет |
| Отчёты для регулятора | ❌ Не реализовано | Формы ПУВА не реализованы |
| Audit trail | ✅ Реализовано | audit_log table, audit routes |

**Формы отчётности ПУВА (нужно реализовать):**

1. **Форма 1-ПУВА:** Ежемесячный отчёт об операциях
   - Количество и объём операций обмена
   - Количество и объём операций торгов
   - Количество активных клиентов
   - Объём активов на хранении (AUC)

2. **Форма 2-ПУВА:** Квартальный отчёт о клиентах
   - Количество зарегистрированных клиентов
   - Количество верифицированных клиентов
   - Распределение по типам (физ.лица / юр.лица)
   - География клиентов

3. **Форма 3-ПУВА:** Отчёт о подозрительных операциях
   - Количество SAR
   - Заблокированные транзакции
   - Высокорисковые адреса

**SQL для генерации отчёта 1-ПУВА:**
```sql
SELECT 
    date_trunc('month', t.created_at) as period,
    o.name as organization,
    COUNT(*) as tx_count,
    SUM(t.amount) as tx_volume,
    COUNT(DISTINCT t.user_id) as active_users,
    SUM(CASE WHEN t.direction = 'incoming' THEN t.amount ELSE 0 END) as inflow,
    SUM(CASE WHEN t.direction = 'outgoing' THEN t.amount ELSE 0 END) as outflow
FROM transactions t
JOIN organizations o ON t.org_id = o.id
WHERE t.created_at >= date_trunc('month', CURRENT_DATE - INTERVAL '1 month')
  AND t.created_at < date_trunc('month', CURRENT_DATE)
GROUP BY 1, 2
ORDER BY 2;
```

## 3. FATF Travel Rule — применимость

### 3.1 Что это
FATF Recommendation 16 (Travel Rule): при переводе ВА на сумму > $1,000 необходимо передать информацию об отправителе и получателе.

### 3.2 Применимость для КР
Кыргызстан — член EAG (Евразийская группа по противодействию отмыванию денег). Travel Rule **применяется** к операторам обмена и торгов ВА.

### 3.3 Что нужно реализовать

| Требование | Статус | Действие |
|-----------|--------|----------|
| Идентификация отправителя | ✅ | Пользователь ORGON = идентифицирован |
| Идентификация получателя | ⚠️ | Для внешних адресов — нужен запрос данных |
| Передача данных VASP→VASP | ❌ | Нужно: интеграция с Travel Rule протоколом (Notabene, Sygna) |
| Хранение данных | ✅ | Все транзакции логируются в БД |

**Рекомендация:** На Phase 3-4 интегрировать Travel Rule протокол (Safina может поддерживать через CCC).
