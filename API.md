# API

The canonical API spec is the live OpenAPI schema generated by FastAPI.

| Surface | URL pattern |
|---|---|
| Swagger UI | `https://<api-host>/api/docs` |
| ReDoc      | `https://<api-host>/api/redoc` |
| OpenAPI JSON | `https://<api-host>/api/openapi.json` |

Concrete `<api-host>` values are tracked in [`DEPLOYMENT.md`](DEPLOYMENT.md)
once the Coolify deploy on the new server is provisioned.

The OpenAPI schema is the **source of truth**. Don't write request/response
shapes in markdown — they will go stale. If you need to publish a contract
externally, snapshot the JSON at a tagged commit.

## Authentication

All authenticated endpoints expect `Authorization: Bearer <jwt>`.

```bash
# Obtain a token
curl -X POST https://<api-host>/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email": "demo-admin@orgon.io", "password": "demo2026"}'

# Use it
curl https://<api-host>/api/dashboard/stats \
  -H "Authorization: Bearer <access_token>"
```

Token lifetime: access 1h, refresh 7d. Refresh via `POST /api/auth/refresh`.

Login is rate-limited to **5 attempts/min per IP** (`/api/auth/login`,
`/api/auth/verify-2fa`, `/api/auth/reset-password`,
`/api/auth/reset-password/confirm`). General API limit: 100 req/min/IP.

`JWT_SECRET_KEY` is required env on prod — the auto-generated fallback
rotates on container restart and silently invalidates every issued token.

## Multi-tenancy

Every authenticated request resolves the user's active `organization_id`
from the JWT. `RLSMiddleware` sets two Postgres session variables
(`app.current_organization_id`, `app.is_super_admin`) at the start of
each request and clears them at the end. RLS policies enforce isolation
at the database level on `wallets`, `transactions`, `signatures`,
`contacts`, `scheduled_transactions`, `audit_log`. The service layer
also filters by `org_ids=[…]` — defense in depth.

`super_admin` users bypass tenant isolation via the
`orgon_current_org_or_super()` SQL helper.

## Partner API (B2B)

`/api/v1/partner/*` endpoints use HMAC-signed requests instead of a JWT.
Required headers per request:

| Header | What |
|---|---|
| `Authorization: Bearer <api_key>` | partner principal — `pk_test_…` or `pk_live_…` |
| `X-Nonce` | unique per request, deduped server-side |
| `X-Timestamp` | Unix seconds, must be within ±5 minutes of server time |
| `X-Signature` | `hmac_sha256_hex(method + "|" + path + "|" + body + "|" + nonce + "|" + timestamp, key=api_secret)` |

Replay protection: `(partner_id, nonce)` is a PK on `partner_request_nonces`.
A 15-minute APScheduler cron prunes nonces older than 1 hour. Drift
beyond ±5 min returns `400 Bad Request`. Replayed nonce returns
`409 Conflict`.

Provisioning: `/api/v1/admin/partners` (super_admin / company_admin):

| Method | Path | Action |
|---|---|---|
| POST | `/api/v1/admin/partners` | create — returns `api_key` and `api_secret` ONCE |
| GET | `/api/v1/admin/partners` | list (scoped to caller's orgs unless super_admin) |
| GET | `/api/v1/admin/partners/{id}` | retrieve single |
| POST | `/api/v1/admin/partners/{id}/rotate` | rotate key — returns new pair ONCE |
| POST | `/api/v1/admin/partners/{id}/revoke` | suspend (`status='suspended'`) |

Partner-org scoping: `partners.organization_id` (nullable) ties an
API-key principal to one organization. Every list/get/cancel call in
`routes_partner*.py` resolves this via `_partner_org_ids(...)` and
passes `org_ids=[…]` into the service layer; cross-tenant lookups
return `404`, never the row from another org. A partner row with
`organization_id IS NULL` sees nothing — that's the safe default.

`ORGON_PARTNER_REPLAY_OFF=1` is a dev-only escape hatch to disable
replay-guard for replay-testing. **Never set on prod.**

## Multi-signature endpoints

`POST /api/v1/partner/transactions/{unid}/sign` and
`POST /api/signatures/{tx_unid}/sign` refuse double-sign / replay attempts:

| Status | Body | When |
|--------|---|------|
| 200    | `{ ok: true, … }`                                              | first time this signer signs this tx |
| 409    | `{ error: "duplicate_signature", message: "…already signed…" }` | same `(tx_unid, signer, action)` row already exists |
| 502    | `{ detail: "..." }`                                            | Safina round-trip failed                |

Implementation: SignatureService checks `signature_history` for an
existing row from `(tx_unid, signer_address)` with
`action IN ('signed', 'rejected')` **before** touching Safina. The
DB-side guarantee is the partial UNIQUE index
`uniq_sig_history_tx_signer_action`.

The legacy `POST /api/transactions/{unid}/{sign,reject}` endpoints have
been **removed** — they bypassed `signature_history` and the replay-guard.

## Billing endpoints

Public:
- `GET /api/v1/billing/plans` — list active subscription plans

Authenticated:
- `GET /api/v1/billing/usage` — usage summary (zeroed default if no
  active subscription, never 404)
- `POST /api/v1/billing/checkout` — create Stripe Checkout Session;
  returns `{url, session_id}` for browser redirect
- `POST /api/v1/billing/payment/manual` — request bank-transfer
  payment for an invoice

Webhook (no auth, HMAC-signed by Stripe):
- `POST /api/v1/billing/webhook` — verifies signature via
  `STRIPE_WEBHOOK_SECRET`, then dispatches:
  - `checkout.session.completed` → `BillingService.upsert_subscription_from_checkout`
  - `customer.subscription.updated|deleted` → mirror status into
    `organization_subscriptions.status`
  - `invoice.payment_failed` → flip status to `past_due`

Handler exceptions are logged + acked (never 5xx) so Stripe doesn't
retry-storm.

Admin (super_admin):
- `GET /api/v1/billing/invoices`, `/payments`
- `POST /api/v1/billing/admin/generate-invoice/{subscription_id}`
- `POST /api/v1/billing/admin/confirm-payment/{payment_id}`

## Webhooks

Outbound: `WebhookService` posts JSON to a partner-registered URL with
an `X-ORGON-Signature: t=<unix>,v1=<hmac-sha256>` header signed with
the per-webhook `secret` (lookup via
`partner_webhooks.secret`; null secret → unsigned, fail-open). Verify
on the partner side before processing.

Inbound: `POST /api/webhooks/safina/callback` for Safina-side events.
HMAC verification using `SAFINA_WEBHOOK_SECRET` `(planned — not yet wired)`.
